Before You Code

Pre-Project Setup
Checklist

The 8 essential steps to complete in your workspace before writing a single line of code. Skipping these is how projects go sideways.

01
📁

Set Up Folder Structure

Organise your project before any files are created. A clear directory layout (assets/, css/, js/, pages/) keeps work scalable and prevents the root-folder chaos that accumulates when you skip this step.

my-project/
├── index.html
├── README.md
├── assets/
│   ├── images/
│   └── fonts/
├── css/
│   └── style.css
└── js/
    └── script.js
OrganisationScalability
02
📝

Create a README.md

Document your project from day one. A good README covers: project name & purpose, features list, how to run it locally, tech stack used, and links to key resources. This is the first file any collaborator — or future you — will read.

  • Project name & description
  • Core features list
  • How to run locally
  • Technologies used
  • Screenshots (add after building)
DocumentationOnboarding
03
🎯

Define the Project Goal

Before touching code, answer these four questions in writing. Vague goals lead to endless rewrites. A clear, written goal acts as a north star when you're tempted to add features that don't serve the core purpose.

  • What problem does this solve?
  • Who is the target audience?
  • What are the v1 core features?
  • What design style? (dark/light, minimal/vibrant)
StrategyScope
04
🎨

Plan Your Design System

Define CSS custom properties (design tokens) at the very top of your stylesheet before writing a single component. This prevents magic numbers from spreading everywhere and ensures visual consistency across every element you build.

:root {
  --color-primary: hsl(250,80%,60%);
  --color-bg: hsl(220,20%,8%);
  --font-main: 'Inter', sans-serif;
  --space-md: 1rem;
  --radius-md: 12px;
  --shadow-md: 0 8px 24px rgba(0,0,0,.3);
}
Design TokensCSS
05
🔤

Choose & Import Fonts

Browser-default fonts signal an unfinished product. Pick a Google Font pair (e.g. Inter + JetBrains Mono) and add the preconnect + stylesheet links to your <head> before writing any CSS. Font choice alone transforms perceived quality.

<link rel="preconnect"
      href="https://fonts.googleapis.com">
<link href="https://fonts.googleapis.com/
csss2?family=Inter:wght@400;600;700
&display=swap" rel="stylesheet">
TypographyBranding
06
🌐

Set Up index.html Properly

Start with a solid HTML5 boilerplate — not a blank file. Include charset, viewport, a descriptive title, meta description, and proper link/script tags. This baseline is the foundation that SEO, accessibility, and performance are all built on.

  • lang attribute on <html>
  • Viewport meta tag set
  • Descriptive <title> tag
  • Meta description for SEO
  • CSS in <head>, JS before </body>
SEOHTML5
07
🔀

Initialise Git

Run git init before writing any code — not after you've already made a mess. Create a .gitignore immediately to keep secrets, node_modules, and OS files out of version history. Commit early and commit often.

git init
git add .
git commit -m "initial project setup"

# .gitignore essentials:
node_modules/
.env
.DS_Store
*.log
GitVersion Control
08
📐

Wireframe the Layout

Sketch each page/screen before writing HTML — even on paper or in Excalidraw. A wireframe reveals layout problems, missing sections, and navigation gaps before you've spent time coding them. It takes 20 minutes and saves hours of restructuring.

  • Map out every section/page
  • Define navigation structure
  • Plan content hierarchy (H1 → H2 → H3)
  • Identify reusable components
  • Plan responsive breakpoints
UXPlanning

GitHub Setup &
Effective Workflow

How to set up GitHub properly, what to do when starting a new project, and how to use GitHub's full feature set to ship better code, faster.

🔧

First-Time GitHub Setup

Do this once on every new machine. These steps connect your local Git to your GitHub account securely.

1

Create a GitHub Account

Go to github.com and sign up. Use a professional username — it appears on your profile, commits, and public repos. Employers and collaborators will see it.

2

Configure Your Identity Locally

Tell Git who you are. This name and email appear on every commit you make.

git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global core.editor "code --wait"  # VS Code as default editor
3

Set Up SSH Authentication

SSH is more secure than HTTPS and avoids typing your password every time. Generate a key and add it to GitHub once.

# Generate SSH key
ssh-keygen -t ed25519 -C "you@example.com"

# Copy the public key to clipboard (Windows)
cat ~/.ssh/id_ed25519.pub | clip

# Then: GitHub → Settings → SSH Keys → New SSH Key → Paste
4

Set Default Branch to “main”

Modern convention uses main instead of master. Set this globally so all new repos start with main.

git config --global init.defaultBranch main
🆕

Starting a New Project on GitHub

The exact sequence to follow the moment you begin a new project. Do this before writing any code.

01

Create Remote Repo on GitHub

  • Go to github.com/new
  • Name it clearly (e.g. rapid-web-dev)
  • Set to Public or Private
  • ✅ Add README, ✅ Add .gitignore, ✅ Choose a licence
  • Click Create repository
02

Clone or Connect Locally

Option A — Clone (recommended for new projects):

git clone git@github.com:username/repo.git
cd repo

Option B — Push existing local project:

git remote add origin git@github.com:username/repo.git
git branch -M main
git push -u origin main
03

Protect the main Branch

Prevent anyone (including yourself) from accidentally force-pushing or committing broken code directly to main.

  • Settings → Branches → Add rule for main
  • ✅ Require pull request reviews before merging
  • ✅ Require status checks to pass (CI)
  • ✅ Disallow force pushes
04

Set Up GitHub Actions CI

Auto-run tests and checks on every push. Create .github/workflows/ci.yml — even a simple lint check adds massive value early.

name: CI
on: [push, pull_request]
jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm run lint
05

Enable GitHub Pages (for static sites)

Deploy your HTML/CSS/JS site for free in seconds — no server needed.

  • Settings → Pages → Source: Deploy from branch
  • Branch: main / folder: / (root)
  • Your site goes live at username.github.io/repo
  • Every push to main auto-deploys
06

Create Project Board & Issues

Use GitHub Projects and Issues as a lightweight Kanban. Create issues for every feature and bug before coding them — this keeps work visible and trackable.

  • Projects → New Project → Board view
  • Columns: Backlog → In Progress → Review → Done
  • Link issues to PRs for automatic close on merge
🌿

Branching Strategy

Never commit directly to main. Use a structured branch naming convention from day one.

Branch Purpose Merges Into
main Production-ready code only. Always deployable. — (protected)
develop Integration branch. All features merge here first. main (via PR)
feature/login-page One branch per feature or task. develop
fix/nav-overflow Bug fixes — small scope, fast turnaround. develop
hotfix/critical-crash Emergency fixes for production issues. main + develop
chore/update-deps Non-feature work: deps, config, tooling, docs. develop
✍️

Commit Message Conventions

Follow Conventional Commits — it makes your history readable, enables changelog generation, and communicates intent clearly.

feat

A new feature added to the app

feat: add user authentication flow
fix

A bug fix — references the issue if possible

fix: resolve nav overflow on mobile (#12)
style

CSS/formatting changes, no logic change

style: update card hover transition timing
refactor

Code restructured, no behaviour change

refactor: extract Button into own component
docs

Documentation only changes

docs: update README with setup instructions
chore

Tooling, deps, config — no source change

chore: update eslint to v9, fix lint errors
test

Adding or updating tests

test: add unit tests for auth middleware
perf

Performance improvements

perf: lazy-load images below the fold

Golden Rules for Commit Messages

✅ Use imperative mood: “add feature” not “added feature”
✅ Keep subject line under 72 characters
✅ Reference issues: “fix: resolve crash (#42)”
✅ One logical change per commit — atomic commits
❌ Don't write “fix stuff” or “WIP” in final commits
❌ Don't commit unrelated changes together
❌ Don't commit broken code to shared branches
❌ Don't force-push to main or develop
⚡

GitHub Features to Use Effectively

Most developers only use 20% of what GitHub offers. Here's the other 80%.

🔀

Pull Requests

Every feature goes through a PR — even when working solo. PRs are your code review gate, your documentation of intent, and your automated test trigger. Write a clear description, link the issue, and add screenshots for UI changes.

💡 Use PR templates (.github/pull_request_template.md) to enforce consistency
📋

Issues & Labels

Treat every bug, feature, and task as an Issue. Use labels (bug, enhancement, priority: high) to organise your backlog. Assign issues to milestones to track version goals.

💡 Close issues automatically: write “Closes #42” in your PR description
⚙️

GitHub Actions

Automate everything: run tests on every PR, lint on push, deploy on merge to main, check for broken links, send Slack notifications. Actions are free for public repos and generous for private ones.

💡 Browse github.com/marketplace for pre-built Actions — don't reinvent the wheel
📄

GitHub Pages

Free hosting for static sites (HTML/CSS/JS, React builds, Jekyll, etc). Perfect for portfolios, documentation, and demo sites. Custom domains supported. Zero server management.

💡 Use gh-pages branch for SPAs so main stays clean
📊

GitHub Projects

A full Kanban/Table/Roadmap board built into GitHub. Link issues and PRs directly to project cards. Use it for sprint planning, feature tracking, and release management — no third-party tool needed.

💡 Use “Iterations” for sprint planning and “Milestones” for release tracking
🔒

Dependabot & Security

Enable Dependabot to auto-create PRs when your dependencies have vulnerabilities or updates. Also enable Code Scanning (GitHub Advanced Security) to catch security issues in your own code.

💡 Settings → Security → Enable all Dependabot features — it's free and saves you from CVEs
🔄

AI-First Development

AI Context Files &
Project Knowledge Management

The secret to 10× better AI output is context. Here’s exactly how to write AGENTS.md, build a docs/ folder, and maintain living documentation so every AI session starts fully informed.

🧠

Without Context Files

AI has zero memory between sessions. You re-explain your stack, naming conventions, folder structure, and decisions every single time. Hours wasted on re-onboarding.

⚡

With Context Files

AI reads your files, instantly understands your architecture, knows your naming conventions, respects your workflow, and produces on-brand code from the very first message of every session.

📝

Writing AGENTS.md (or claude.md) from Day 1

Create this file in your project root on the very first day. It is the AI’s permanent briefing document — read automatically at the start of every session. Write it like you’re onboarding a new senior developer.

What to Include

🎯 Project Overview
🛠️ Tech Stack
📁 Folder Structure
🎨 Design System

✏️ Commit Convention

# AGENTS.md — AI Assistant Briefing
# Read this before doing anything in this project.

## Project Overview
- Name: MyApp — A SaaS dashboard for freelancers
- Goal: Help freelancers track projects, invoices, and time
- Status: MVP in progress — targeting v1.0 by Aug 2025
- Live URL: https://myapp.vercel.app (staging)

## Tech Stack
- Frontend: Next.js 14 (App Router), TypeScript, Tailwind CSS
- Backend: Next.js API routes + Prisma ORM
- Database: PostgreSQL (Neon serverless)
- Auth: NextAuth.js v5 (Google + email)
- Deploy: Vercel (preview + production)
- Testing: Vitest + Playwright

## Folder Structure
src/
├── app/              ← Next.js App Router pages
├── components/       ← Reusable UI components
│   ├── ui/           ← Primitive components (Button, Input...)
│   └── features/     ← Feature-specific components
├── lib/              ← Utilities, helpers, constants
├── hooks/            ← Custom React hooks
├── types/            ← TypeScript type definitions
├── server/           ← Server actions + API logic
└── styles/           ← Global CSS, design tokens

## Design System & Conventions
- Colors: Use Tailwind classes only, never hardcode hex
- Spacing: Always use Tailwind spacing scale (p-4, gap-6...)
- Components: Use shadcn/ui as base, extend in components/ui/
- Icons: lucide-react ONLY, no other icon libraries
- Fonts: Inter (body), JetBrains Mono (code)

## Coding Standards
- TypeScript strict mode is ON — no 'any' types allowed
- Components use named exports, never default exports
- File naming: kebab-case for files, PascalCase for components
- All server actions live in src/server/actions/
- API errors must use our custom ApiError class (see lib/errors.ts)
- Always handle loading and error states in UI components

## Git Workflow
- Branch from: develop (never main directly)
- Branch naming: feature/, fix/, chore/, hotfix/
- Commit style: Conventional Commits (feat:, fix:, docs:...)
- PRs require: passing CI + 1 review before merge
- Never force-push to main or develop

## What NOT to Do
- Do NOT install new npm packages without asking first
- Do NOT modify the Prisma schema without discussing migration plan
- Do NOT change NextAuth config — it is intentionally complex
- Do NOT use 'any' type in TypeScript
- Do NOT hardcode strings — use constants from lib/constants.ts
- Do NOT commit .env files or secrets

## Current Sprint Focus
- Building: Invoice generation feature (see docs/features/invoices.md)
- Blocked on: Stripe webhook integration (see issue #47)
- Next up: PDF export + email delivery

## Important Files to Know
- lib/auth.ts — auth configuration and session helpers
- lib/db.ts — Prisma client singleton
- lib/constants.ts — all magic strings and enums
- components/ui/Button.tsx — always use this, never <button>
- docs/ARCHITECTURE.md — system design and data flow
📁

The docs/ Folder — Your Project’s Brain

Maintain a docs/ folder at the project root. This is where all architecture decisions, feature specs, and technical notes live. Reference these from AGENTS.md so AI can read them when needed.

ARCHITECTURE.md

System Architecture

High-level system diagram, data flow, service boundaries. How frontend talks to backend, what the database schema looks like at a macro level, any external APIs integrated.

Update whenever you add a new service or major data model
DECISIONS.md

Architecture Decision Records (ADRs)

Log every major technical decision with: Context (why was this decision needed?), Options considered, Decision made, and Consequences. Invaluable for AI and future-you.

Example: “Chose Prisma over Drizzle because…”
features/[name].md

Feature Specifications

One file per feature. Include: user story, acceptance criteria, UI mockup link, API endpoints affected, data model changes, edge cases to handle. AI uses this to implement features correctly on the first try.

Write the spec before opening a ticket or writing code
API.md

API Reference

Document every endpoint: method, URL, request body schema, response shape, auth requirements, error codes. AI uses this to write accurate API calls, types, and integration tests without guessing.

Auto-generate with tools like Swagger/OpenAPI, then export to markdown
PROGRESS.md

Living Progress Log

A running log of what was built, what was decided, and what’s left. Update at the end of every session. This is the most powerful file for AI continuity — it tells the AI exactly where things stand.

Ask Antigravity to update this file at the end of every session
TROUBLESHOOTING.md

Known Issues & Gotchas

Document every tricky bug, unexpected behaviour, or environment quirk you’ve discovered. Prevents AI (and you) from re-solving the same problem. Include the symptom, root cause, and fix.

Add an entry every time you spend >30 mins on a non-obvious problem

Keeping Context Files Alive

A stale AGENTS.md is worse than none — it misleads the AI. Follow these rules to keep your context files current without it becoming a burden.

✅

Update AGENTS.md on every stack change

Added a new library? Changed a naming convention? Switched auth providers? Update AGENTS.md in the same commit. Treat it like updating package.json.

✅

Ask AI to update PROGRESS.md at session end

At the end of every work session, prompt: “Update docs/PROGRESS.md with what we built today, key decisions made, and what’s left for next session.” This gives future sessions instant context.

✅

Write feature specs before code

Before asking AI to build a feature, create docs/features/feature-name.md first. Include the user story, acceptance criteria, and edge cases. Then say: “Build this per the spec in docs/features/x.md.”

✅

Log every ADR immediately

The moment you make a technical decision (“we’re using Zustand instead of Redux because…”), add it to docs/DECISIONS.md. Future AI sessions won’t undo your decisions if they’re documented.

❌

Don’t let AGENTS.md become a monolith

Keep AGENTS.md to 200–400 lines max. Move deep feature details to docs/features/. Reference those files from AGENTS.md instead of copying content in. AI can read linked files on demand.

❌

Don’t document the obvious

Don’t write “useState manages local state” in AGENTS.md. Document what’s non-obvious, project-specific, or against framework defaults. Focus on YOUR conventions, not general React knowledge.

🤝

Session Handoff Template

Ask AI to fill this template at the end of every session. Paste it as the first message of your next session to instantly reload full context without repeating yourself.

[paste the 3-line summary here]

Ship It Right

Pre-Launch
Checklist

The non-negotiable things to verify before your project goes live. One missed item can cost you users, SEO rank, or reputation.

🔍

SEO & Meta

  • Unique <title> tag on every page (<60 chars)
  • Meta description written (150–160 chars)
  • Open Graph tags: og:title, og:description, og:image, og:url
  • Twitter Card meta tags set
  • Canonical URL set to avoid duplicate content
  • robots.txt created and correct
  • sitemap.xml generated and submitted to Google Search Console
  • Structured data (JSON-LD) added where relevant
⚡

Performance

  • Lighthouse score ≥ 90 on all 4 metrics
  • LCP < 2.5s, FID < 100ms, CLS < 0.1
  • All images compressed & in WebP format
  • Images have width & height to prevent CLS
  • Fonts preloaded with rel="preconnect"
  • JS bundle < 150KB gzipped for initial load
  • CSS minified, unused CSS purged
  • Browser caching configured (Cache-Control headers)
♿

Accessibility

  • All images have descriptive alt text
  • Colour contrast ≥ 4.5:1 for normal text (WCAG AA)
  • Entire app is keyboard-navigable (Tab through it)
  • Visible focus indicators on all interactive elements
  • ARIA labels on buttons with no text content
  • Skip navigation link for screen readers
  • Heading hierarchy correct (one H1, logical H2→H3)
  • Run axe DevTools or WAVE — zero violations
🔒

Security

  • HTTPS enforced, HSTS header set
  • Content-Security-Policy header configured
  • All user inputs sanitised server-side
  • No secrets or API keys in client-side code or Git history
  • Authentication tokens in httpOnly cookies
  • Rate limiting on auth endpoints
  • npm audit passes with zero high-severity issues
  • X-Content-Type-Options + X-Frame-Options headers set
📱

Cross-Device Testing

  • Tested on Chrome, Firefox, Safari, Edge
  • Tested on iOS Safari (real device or BrowserStack)
  • Tested on Android Chrome
  • Responsive at 320px, 375px, 768px, 1024px, 1440px
  • Touch targets ≥ 44×44px on mobile
  • No horizontal scrollbar at any viewport
  • Forms work correctly on mobile keyboards
📊

Monitoring & Analytics

  • Error tracking active (Sentry, LogRocket, or similar)
  • Analytics set up (Google Analytics, Plausible, Umami)
  • Uptime monitoring configured (UptimeRobot or Better Uptime)
  • Custom 404 page designed and working
  • Custom 500 error page in place
  • All environment variables set in production
  • Rollback plan documented and tested
Styling Mastery

CSS Architecture &
Responsive Design

How to structure CSS that scales, write mobile-first layouts that don’t break, and use the cascade without fighting it.

🎨

Design Tokens First

Always define all colours, spacing, radius, shadows, and typography as CSS custom properties in :root before writing a single component. This is your single source of truth.

:root {
  --clr-primary: hsl(250 80% 60%);
  --space-4: 1rem;
  --radius-md: 12px;
  --text-body: 'Inter', sans-serif;
}
📱

Mobile-First Media Queries

Write base styles for mobile, then use min-width to scale up. Never use max-width to undo mobile styles — that’s desktop-first thinking and creates specificity chaos.

/* Mobile base (default) */
.card { padding: 1rem; }

/* Tablet and up */
@media (min-width: 768px) {
  .card { padding: 2rem; }
}

/* Desktop and up */
@media (min-width: 1200px) {
  .card { padding: 3rem; }
}
📌

Flexbox vs Grid — The Rule

Flexbox = 1-dimensional. Use for: navbars, button groups, centering a single item, card content layout. Grid = 2-dimensional. Use for: page layouts, card grids, anything with rows AND columns.

/* Flex — 1D row alignment */
.nav { display: flex; gap: 1rem; align-items: center; }

/* Grid — 2D card layout */
.cards {
  display: grid;
  grid-template-columns:
    repeat(auto-fill, minmax(300px, 1fr));
  gap: 1.5rem;
}
📝

BEM Naming Convention

Block__Element--Modifier. Name classes so they describe structure, not style. A class named .btn--primary tells you what it is. A class named .blue-large tells you nothing useful.

/* Block */
.card { }
/* Element */
.card__header { }
.card__body { }
/* Modifier */
.card--featured { }
.card--compact { }
📏

Fluid Typography with clamp()

Use clamp(min, preferred, max) to create font sizes that scale fluidly between breakpoints without any media queries. Set it once, forget it.

/* Scales from 1.25rem to 3rem */
h1 {
  font-size: clamp(1.25rem, 5vw, 3rem);
}

/* Container padding that adapts */
.container {
  padding-inline:
    clamp(1rem, 5vw, 3rem);
}
🧩

The Box Model — border-box

Always reset to border-box globally. With the default content-box, adding padding makes elements larger than intended. This is the #1 source of layout bugs for beginners.

/* Always put this at the top of your CSS */
*, *::before, *::after {
  box-sizing: border-box;
}

/* Now padding/border are INSIDE the width */
.box {
  width: 200px;
  padding: 20px; /* doesn't expand the box */
}
⚡

CSS Specificity Order

Specificity determines which rule wins when two selectors target the same element. Understanding it prevents the !important spiral that destroys codebases.

/* Lowest → Highest specificity */
* { }              /* 0,0,0 */
element { }        /* 0,0,1 */
.class { }         /* 0,1,0 */
#id { }            /* 1,0,0 */
style="..."        /* inline */
!important         /* nuclear option */

/* Keep specificity flat — all classes */
🎥

Smooth Animations

Animate only transform and opacity for 60fps performance. Animating width, height, top, or margin triggers expensive layout reflow. Use will-change sparingly.

/* ✅ GPU-accelerated */
.card:hover {
  transform: translateY(-4px);
  opacity: 0.9;
  transition: transform 0.3s, opacity 0.3s;
}

/* ❌ Triggers layout */
.card:hover { top: -4px; width: 102%; }

Performance
Optimization Guide

Every 100ms of delay costs ~1% conversion. Here’s exactly how to make your web app fast — measured, not guessed.

LCP Largest Contentful Paint < 2.5s ✓
CLS Cumulative Layout Shift < 0.1 ✓
INP Interaction to Next Paint < 200ms ✓
TTFB Time to First Byte < 800ms ✓
🖼️

Images

  • Use WebP format (30–50% smaller than PNG/JPG)
  • Always set width & height to prevent CLS
  • Add loading="lazy" for below-fold images
  • Use srcset + sizes for responsive images
  • Serve via CDN (Cloudinary, Imgix, or Vercel Image)
  • Use decoding="async" on non-critical images
📜

JavaScript

  • Lazy-load routes and large components (dynamic import)
  • Tree-shake unused code — never import entire libraries
  • Defer non-critical scripts with defer or async
  • Bundle size: keep initial JS < 150KB gzipped
  • Analyse with Webpack Bundle Analyzer or vite --analyze
  • Move heavy computation to Web Workers
🎨

CSS

  • Inline critical CSS in <head> for above-fold content
  • Load non-critical CSS asynchronously
  • Use font-display: swap on web fonts
  • Preconnect to font CDN: rel="preconnect"
  • Purge unused CSS (PurgeCSS, Tailwind purge)
  • Avoid large CSS files > 50KB uncompressed
📦

Caching & CDN

  • Serve static assets via CDN edge nodes globally
  • Long-term cache: Cache-Control: max-age=31536000, immutable
  • Use content hashing in filenames for cache busting
  • Cache API responses with proper ETags
  • Enable Brotli or Gzip compression on server
  • Use a service worker for offline caching
🔧

Server & Network

  • Reduce TTFB: use edge functions or SSR caching
  • HTTP/2 or HTTP/3 — enables multiplexing
  • Preload key resources: <link rel="preload">
  • Use dns-prefetch for external domains
  • Minimise redirects — each adds 100–300ms
  • Use Resource Hints: prefetch next-page resources
📊

Measure First

  • Run Lighthouse in Chrome DevTools before optimising
  • Use WebPageTest.org for real-world waterfall analysis
  • Monitor with Google Search Console CWV report
  • Profile JS with Performance tab in DevTools
  • Never optimise without measuring first — guessing is waste
  • Set a performance budget and enforce it in CI
Quality Assurance

Testing
Strategy

Not everything needs a test. But critical paths must never break silently. Here’s what to test, how, and when.

Few, Slow, Expensive
E2E Tests

Playwright or Cypress — test complete user journeys (login → checkout → success). Slow but catches integration failures. Run on CI before every deploy. Aim for 5–20 critical path tests.

Integration Tests

Test how components work together: API routes, database queries, auth middleware. Use Supertest for HTTP. More than unit tests, fewer than E2E. Fast enough to run on every PR.

Unit Tests

Test individual functions and components in isolation. Use Vitest or Jest. Should be the majority of your test suite. Run in milliseconds. Cover: utilities, validation, business logic, edge cases.

Many, Fast, Cheap

🎯 What to Test

  • All business logic and utility functions
  • Auth flows — login, logout, session expiry
  • Payment & checkout critical paths
  • Form validation (valid + invalid inputs)
  • API endpoints: success, error, and edge cases
  • Components that change based on state/props

🚫 What NOT to Test

  • Third-party libraries (they have their own tests)
  • Implementation details — test behaviour, not internals
  • Simple getters/setters with no logic
  • Static HTML markup that never changes
  • Config files and constants
  • Code that will definitely be deleted soon

✅ Testing Principles

  • Write tests before fixing bugs — reproduce first
  • One assertion per test when possible
  • Tests should be readable like documentation
  • Fast feedback: unit tests in < 1s total
  • Aim for 80% coverage on critical modules
  • Tests that always pass without asserting are useless

API Design
Best Practices

A well-designed API is a product. These conventions make your API predictable, maintainable, and a joy to consume.

🔗 REST Conventions

GET    /users          ← List all
GET    /users/:id      ← Get one
POST   /users          ← Create
PUT    /users/:id      ← Full replace
PATCH  /users/:id      ← Partial update
DELETE /users/:id      ← Delete

✅ Plural nouns, never verbs
❌ /getUser  /createInvoice

🟢 HTTP Status Codes

200 OK          ← Success
201 Created     ← POST success
204 No Content  ← DELETE success
400 Bad Request ← Invalid input
401 Unauthorized← Not authenticated
403 Forbidden   ← Not authorized
404 Not Found   ← Resource missing
409 Conflict    ← Duplicate data
422 Unprocessable← Validation fail
429 Too Many Req← Rate limited
500 Server Error← Our fault

📦 Response Shape

Always use a consistent envelope structure:

// Success
{
  "data": { ... },
  "meta": { "page": 1, "total": 42 }
}

// Error
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Email is required",
    "field": "email"
  }
}

📌 API Design Rules

  • Version your API from day 1: /api/v1/
  • Never return raw DB errors to the client
  • Validate all inputs server-side, always
  • Use pagination for any list endpoint
  • Return the created/updated resource in the response
  • Be idempotent: same request = same result
  • Document every endpoint before building it
  • Rate limit all public endpoints

🔐 Authentication Patterns

  • Session cookies — best for web apps, httpOnly + SameSite=Strict
  • JWT — stateless, for APIs & mobile; use short expiry + refresh tokens
  • API Keys — for server-to-server; never expose in frontend code
  • OAuth 2.0 — for third-party login (Google, GitHub)
  • Always send auth tokens in headers, never in URLs
  • Rotate secrets regularly; revoke on compromise immediately

⚡ API Performance

  • Profile every DB query — watch for N+1 problems
  • Add database indexes on frequently queried fields
  • Cache expensive queries with Redis or in-memory cache
  • Use cursor-based pagination for large datasets
  • Compress responses with Brotli/Gzip
  • Set timeouts on all external API calls

The 12 Universal Laws of
Web App Development

These principles apply regardless of your framework, language, or stack. Violate them and you'll pay the price later.

01
🏗️

Separation of Concerns

Keep HTML (structure), CSS (style), and JS (behaviour) cleanly separated. Each file/module should do one thing well. Mixing them creates a maintenance nightmare that compounds over time.

ArchitectureMaintainability
02
📱

Mobile-First, Always

Design and code for the smallest viewport first, then scale up. Over 60% of web traffic is mobile. Starting desktop-first and squishing down is a recipe for broken layouts and poor UX.

ResponsiveUX
03
⚡

Performance IS a Feature

Every 100ms delay reduces conversion by ~1%. Users abandon pages that take >3s. Lighthouse score, bundle size, and Core Web Vitals are not optional metrics — they are product requirements.

PerformanceCWV
04
♿

Accessibility is Non-Negotiable

WCAG compliance is both a legal requirement and the right thing to do. Use semantic HTML, ARIA roles, sufficient colour contrast, and keyboard navigability. Screen readers exist for real users.

A11yWCAG
05
🔒

Security by Design

Never trust user input. Sanitise everything server-side. Use HTTPS everywhere, implement CSP headers, avoid XSS and CSRF vectors, and never store secrets in client code. Security is never an afterthought.

SecurityOWASP
06
🧩

Component-Based Thinking

Build the UI as a tree of reusable, composable components. Each component owns its state, markup, and styles. This scales far better than monolithic pages and enables parallel development.

ArchitectureReusability
07
🗄️

State is the Source of Truth

UI is a function of state. Manage state at the right layer — local vs global vs server state. Avoid duplicating state; when state diverges from UI you get bugs that are nearly impossible to reproduce.

State MgmtData Flow
08
🔗

API-First Architecture

Treat your backend as a product, not an implementation detail. Design clear, versioned APIs before writing a line of UI. This decouples frontend from backend and enables parallel development.

API DesignDecoupling
09
🧪

Test What Matters

Not everything needs a unit test, but critical paths must be tested. Follow the Testing Pyramid: many unit tests, fewer integration tests, few end-to-end tests. Untested code is technical debt waiting to explode.

TestingQuality
10
🚀

CI/CD is Table Stakes

Automate building, testing, and deploying from day one. Manual deployments breed fear and slow velocity. A broken CI pipeline should stop all other work — it's your quality gate.

DevOpsAutomation
11
📐

Progressive Enhancement

Build a working baseline first (pure HTML), then layer in CSS enhancements, then JS interactivity. This ensures your app works for every user on every device, even with JS disabled or slow connections.

ResilienceUX
12

Measure Before You Optimise

Premature optimisation is the root of all evil. Profile first, identify the real bottleneck, then optimise surgically. Optimising the wrong thing wastes time and often makes code worse.

Self-Awareness

Critical Knowledge Gaps
to Fill Before You Build

Honest blind spots that trip up most developers. Knowing what you don't know is the first step to mastery.

🌐

Networking & HTTP

Critical

HTTP Methods Semantics

GET is idempotent & safe. POST creates. PUT replaces. PATCH partially updates. DELETE removes. Misusing them breaks REST contracts and caching.

Important

CORS — How It Really Works

CORS is enforced by the browser, not the server. The server only sends headers. Understanding preflight requests saves hours of debugging.

Important

HTTP Caching Headers

Cache-Control, ETag, Last-Modified. Proper caching = faster apps with less server load. Getting it wrong = stale data reaching users.

🎨

CSS Fundamentals

Critical

The Cascade & Specificity

Specificity wars cause "why isn't my style applying?" panic. Understanding the cascade (origin → specificity → order) fixes this permanently.

Critical

CSS Box Model

content-box vs border-box. Margins collapse. Padding vs margin. Most layout bugs trace back to misunderstanding the box model.

Important

Flexbox vs Grid — When to Use Which

Flex is 1D (row OR column). Grid is 2D. Use Grid for page layout, Flex for component-level alignment.

⚡

JavaScript Internals

Critical

Event Loop & Async/Await

JS is single-threaded. The event loop enables concurrency. Not understanding it leads to race conditions, blocking the UI thread, and missed error handling.

Critical

Closures & Scope

Closures are how JS handles private state. Misunderstanding scope causes "variable is undefined" bugs and stale closure bugs in React hooks.

Good to Know

Prototype Chain & this

Understanding this binding prevents a class of bugs that are notoriously confusing, especially in callbacks and event handlers.

🔒

Security Essentials

Critical

XSS (Cross-Site Scripting)

Injecting malicious scripts via user input. Never use innerHTML with user data. Sanitise all inputs. Use a Content Security Policy.

Critical

CSRF (Cross-Site Request Forgery)

Tricking authenticated users into submitting unintended requests. Defend with CSRF tokens and SameSite cookie attribute.

Important

JWT Authentication Pitfalls

JWTs are not sessions. They can't be invalidated server-side until expiry. Use short expiry + refresh tokens. Never store in localStorage.

🗄️

Database & State

Critical

N+1 Query Problem

Fetching a list of N items then making N additional queries for each item. This silently kills performance. Always profile your database queries.

Important

SQL vs NoSQL Trade-offs

Relational DBs for structured, consistent data. NoSQL for flexible schemas, horizontal scale. Choosing wrong is painful to undo later.

Important

Optimistic vs Pessimistic Updates

Update the UI immediately (optimistic) vs wait for server confirmation (pessimistic). Each has trade-offs in UX and data consistency.

📦

Build & Deployment

Critical

Environment Variables & Secrets

Never hardcode secrets. Use .env files locally, proper secret managers in production (Vault, AWS Secrets Manager). Leaked secrets = critical vulnerability.

Important

Bundle Size & Code Splitting

Shipping a 5MB JS bundle tanks your performance. Use lazy loading, tree shaking, and code splitting. Analyse your bundle with Webpack Bundle Analyzer.

Good to Know

CDN & Edge Caching

Serving assets from geographically close CDN nodes drastically reduces latency. Misconfigured CDN cache means users see stale content after deploys.

  • Content-Security-Policy header configured
  • All user inputs sanitised server-side
  • No secrets in client-side code or git
  • Authentication tokens use httpOnly cookies
  • Rate limiting on all auth endpoints
  • Dependencies updated (npm audit passing)

  • Screen reader tested (NVDA / VoiceOver)
  • No reliance on colour alone for meaning
