Before You Code

Pre-Project Setup
Checklist

The 8 essential steps to complete in your workspace before writing a single line of code. Skipping these is how projects go sideways.

01
📁

Set Up Folder Structure

Organise your project before any files are created. A clear directory layout (assets/, css/, js/, pages/) keeps work scalable and prevents the root-folder chaos that accumulates when you skip this step.

my-project/
├── index.html
├── README.md
├── assets/
│   ├── images/
│   └── fonts/
├── css/
│   └── style.css
└── js/
    └── script.js
OrganisationScalability
02
📝

Create a README.md

Document your project from day one. A good README covers: project name & purpose, features list, how to run it locally, tech stack used, and links to key resources. This is the first file any collaborator — or future you — will read.

  • Project name & description
  • Core features list
  • How to run locally
  • Technologies used
  • Screenshots (add after building)
DocumentationOnboarding
03
🎯

Define the Project Goal

Before touching code, answer these four questions in writing. Vague goals lead to endless rewrites. A clear, written goal acts as a north star when you're tempted to add features that don't serve the core purpose.

  • What problem does this solve?
  • Who is the target audience?
  • What are the v1 core features?
  • What design style? (dark/light, minimal/vibrant)
StrategyScope
04
🎨

Plan Your Design System

Define CSS custom properties (design tokens) at the very top of your stylesheet before writing a single component. This prevents magic numbers from spreading everywhere and ensures visual consistency across every element you build.

:root {
  --color-primary: hsl(250,80%,60%);
  --color-bg: hsl(220,20%,8%);
  --font-main: 'Inter', sans-serif;
  --space-md: 1rem;
  --radius-md: 12px;
  --shadow-md: 0 8px 24px rgba(0,0,0,.3);
}
Design TokensCSS
05
🔤

Choose & Import Fonts

Browser-default fonts signal an unfinished product. Pick a Google Font pair (e.g. Inter + JetBrains Mono) and add the preconnect + stylesheet links to your <head> before writing any CSS. Font choice alone transforms perceived quality.

<link rel="preconnect"
      href="https://fonts.googleapis.com">
<link href="https://fonts.googleapis.com/
csss2?family=Inter:wght@400;600;700
&display=swap" rel="stylesheet">
TypographyBranding
06
🌐

Set Up index.html Properly

Start with a solid HTML5 boilerplate — not a blank file. Include charset, viewport, a descriptive title, meta description, and proper link/script tags. This baseline is the foundation that SEO, accessibility, and performance are all built on.

  • lang attribute on <html>
  • Viewport meta tag set
  • Descriptive <title> tag
  • Meta description for SEO
  • CSS in <head>, JS before </body>
SEOHTML5
07
🔀

Initialise Git

Run git init before writing any code — not after you've already made a mess. Create a .gitignore immediately to keep secrets, node_modules, and OS files out of version history. Commit early and commit often.

git init
git add .
git commit -m "initial project setup"

# .gitignore essentials:
node_modules/
.env
.DS_Store
*.log
GitVersion Control
08
📐

Wireframe the Layout

Sketch each page/screen before writing HTML — even on paper or in Excalidraw. A wireframe reveals layout problems, missing sections, and navigation gaps before you've spent time coding them. It takes 20 minutes and saves hours of restructuring.

  • Map out every section/page
  • Define navigation structure
  • Plan content hierarchy (H1 → H2 → H3)
  • Identify reusable components
  • Plan responsive breakpoints
UXPlanning

The 12 Universal Laws of
Web App Development

These principles apply regardless of your framework, language, or stack. Violate them and you'll pay the price later.

01
🏗️

Separation of Concerns

Keep HTML (structure), CSS (style), and JS (behaviour) cleanly separated. Each file/module should do one thing well. Mixing them creates a maintenance nightmare that compounds over time.

ArchitectureMaintainability
02
📱

Mobile-First, Always

Design and code for the smallest viewport first, then scale up. Over 60% of web traffic is mobile. Starting desktop-first and squishing down is a recipe for broken layouts and poor UX.

ResponsiveUX
03
⚡

Performance IS a Feature

Every 100ms delay reduces conversion by ~1%. Users abandon pages that take >3s. Lighthouse score, bundle size, and Core Web Vitals are not optional metrics — they are product requirements.

PerformanceCWV
04
♿

Accessibility is Non-Negotiable

WCAG compliance is both a legal requirement and the right thing to do. Use semantic HTML, ARIA roles, sufficient colour contrast, and keyboard navigability. Screen readers exist for real users.

A11yWCAG
05
🔒

Security by Design

Never trust user input. Sanitise everything server-side. Use HTTPS everywhere, implement CSP headers, avoid XSS and CSRF vectors, and never store secrets in client code. Security is never an afterthought.

SecurityOWASP
06
🧩

Component-Based Thinking

Build the UI as a tree of reusable, composable components. Each component owns its state, markup, and styles. This scales far better than monolithic pages and enables parallel development.

ArchitectureReusability
07
🗄️

State is the Source of Truth

UI is a function of state. Manage state at the right layer — local vs global vs server state. Avoid duplicating state; when state diverges from UI you get bugs that are nearly impossible to reproduce.

State MgmtData Flow
08
🔗

API-First Architecture

Treat your backend as a product, not an implementation detail. Design clear, versioned APIs before writing a line of UI. This decouples frontend from backend and enables parallel development.

API DesignDecoupling
09
🧪

Test What Matters

Not everything needs a unit test, but critical paths must be tested. Follow the Testing Pyramid: many unit tests, fewer integration tests, few end-to-end tests. Untested code is technical debt waiting to explode.

TestingQuality
10
🚀

CI/CD is Table Stakes

Automate building, testing, and deploying from day one. Manual deployments breed fear and slow velocity. A broken CI pipeline should stop all other work — it's your quality gate.

DevOpsAutomation
11
📐

Progressive Enhancement

Build a working baseline first (pure HTML), then layer in CSS enhancements, then JS interactivity. This ensures your app works for every user on every device, even with JS disabled or slow connections.

ResilienceUX
12
📊

Measure Before You Optimise

Premature optimisation is the root of all evil. Profile first, identify the real bottleneck, then optimise surgically. Optimising the wrong thing wastes time and often makes code worse.

PerformanceProfiling
Process

The 7-Phase
Development Workflow

A repeatable, battle-tested process that works for solo devs and large teams alike.

Phase 1
🎯

Discovery & Requirements

Define the problem, not the solution. Interview stakeholders, document user stories, set measurable acceptance criteria. Ambiguous requirements are the #1 cause of project failure.

  • User stories & personas defined
  • Acceptance criteria written
  • Tech constraints identified
  • Success metrics agreed upon
📄 Output: PRD / Spec Doc
Phase 2
🗺️

Architecture & Design

Choose your tech stack deliberately. Design the system architecture, data models, and API contracts. Create wireframes and high-fidelity mockups. Validate with stakeholders before writing code.

  • System diagram created
  • Data models defined
  • API contracts documented
  • UI/UX mockups approved
📐 Output: Architecture Doc + Mockups
Phase 3
🏗️

Environment Setup & Foundation

Set up version control, CI/CD pipeline, dev/staging/prod environments, linting, formatting, and the project scaffold. A solid foundation prevents technical debt from accumulating.

  • Git branching strategy set
  • CI/CD pipeline configured
  • Environment variables managed
  • Code style enforced (ESLint/Prettier)
⚙️ Output: Runnable scaffold + CI pipeline
Phase 4
💻

Core Development

Build features in vertical slices (full-stack per feature), not horizontal layers (all backend then all frontend). Use feature flags for incomplete work. Commit small, deploy often.

  • Feature-by-feature development
  • Unit tests written alongside code
  • Code reviews for every PR
  • No broken builds merged
🛠️ Output: Working feature increments
Phase 5
🧪

Testing & QA

Run the full testing suite — unit, integration, end-to-end, accessibility, performance, and security audits. Cross-browser and cross-device testing. Fix bugs before adding more features.

  • Unit + integration tests passing
  • E2E tests covering critical paths
  • Lighthouse score ≥ 90
  • Zero WCAG AA violations
✅ Output: Test report + bug fixes
Phase 6
🚀

Deployment & Launch

Deploy to production via your automated pipeline. Use blue-green or canary releases to minimise risk. Set up monitoring, error tracking, and alerting before launch, not after.

  • Production deploy automated
  • Error tracking active (Sentry/etc.)
  • Uptime monitoring configured
  • Rollback plan documented
🌐 Output: Live production app
Phase 7
📈

Iterate & Optimise

Analyse real user data, collect feedback, measure against success metrics, and prioritise the next iteration. Software is never "done" — it evolves in response to real-world usage.

  • Analytics reviewed weekly
  • User feedback collected
  • Performance monitored
  • Backlog prioritised
🔄 Output: Next iteration plan
Self-Awareness

Critical Knowledge Gaps
to Fill Before You Build

Honest blind spots that trip up most developers. Knowing what you don't know is the first step to mastery.

🌐

Networking & HTTP

Critical

HTTP Methods Semantics

GET is idempotent & safe. POST creates. PUT replaces. PATCH partially updates. DELETE removes. Misusing them breaks REST contracts and caching.

Important

CORS — How It Really Works

CORS is enforced by the browser, not the server. The server only sends headers. Understanding preflight requests saves hours of debugging.

Important

HTTP Caching Headers

Cache-Control, ETag, Last-Modified. Proper caching = faster apps with less server load. Getting it wrong = stale data reaching users.

🎨

CSS Fundamentals

Critical

The Cascade & Specificity

Specificity wars cause "why isn't my style applying?" panic. Understanding the cascade (origin → specificity → order) fixes this permanently.

Critical

CSS Box Model

content-box vs border-box. Margins collapse. Padding vs margin. Most layout bugs trace back to misunderstanding the box model.

Important

Flexbox vs Grid — When to Use Which

Flex is 1D (row OR column). Grid is 2D. Use Grid for page layout, Flex for component-level alignment.

⚡

JavaScript Internals

Critical

Event Loop & Async/Await

JS is single-threaded. The event loop enables concurrency. Not understanding it leads to race conditions, blocking the UI thread, and missed error handling.

Critical

Closures & Scope

Closures are how JS handles private state. Misunderstanding scope causes "variable is undefined" bugs and stale closure bugs in React hooks.

Good to Know

Prototype Chain & this

Understanding this binding prevents a class of bugs that are notoriously confusing, especially in callbacks and event handlers.

🔒

Security Essentials

Critical

XSS (Cross-Site Scripting)

Injecting malicious scripts via user input. Never use innerHTML with user data. Sanitise all inputs. Use a Content Security Policy.

Critical

CSRF (Cross-Site Request Forgery)

Tricking authenticated users into submitting unintended requests. Defend with CSRF tokens and SameSite cookie attribute.

Important

JWT Authentication Pitfalls

JWTs are not sessions. They can't be invalidated server-side until expiry. Use short expiry + refresh tokens. Never store in localStorage.

🗄️

Database & State

Critical

N+1 Query Problem

Fetching a list of N items then making N additional queries for each item. This silently kills performance. Always profile your database queries.

Important

SQL vs NoSQL Trade-offs

Relational DBs for structured, consistent data. NoSQL for flexible schemas, horizontal scale. Choosing wrong is painful to undo later.

Important

Optimistic vs Pessimistic Updates

Update the UI immediately (optimistic) vs wait for server confirmation (pessimistic). Each has trade-offs in UX and data consistency.

📦

Build & Deployment

Critical

Environment Variables & Secrets

Never hardcode secrets. Use .env files locally, proper secret managers in production (Vault, AWS Secrets Manager). Leaked secrets = critical vulnerability.

Important

Bundle Size & Code Splitting

Shipping a 5MB JS bundle tanks your performance. Use lazy loading, tree shaking, and code splitting. Analyse your bundle with Webpack Bundle Analyzer.

Good to Know

CDN & Edge Caching

Serving assets from geographically close CDN nodes drastically reduces latency. Misconfigured CDN cache means users see stale content after deploys.

Antigravity is a powerful agentic AI coding assistant. Here's how to unlock its full potential and use it at its absolute best.

⬡

What is Antigravity?

Antigravity is an AI coding agent powered by Google DeepMind's advanced models. Unlike a simple chatbot, it's an autonomous agent — it can read and write files, run commands in your terminal, browse the web, search your codebase, and even spawn sub-agents to tackle tasks in parallel. Think of it as a senior engineer on your team who never sleeps and works at machine speed.

01

Give Context, Not Commands

The more context you provide, the better the output. Don't just say "build a login page." Say: "Build a login page for a SaaS app. Users log in with email/password. It should use our existing dark theme (see style.css), handle errors gracefully, and redirect to /dashboard on success."

💡 Tip: Include the WHY, not just the WHAT. "I need this to be fast because it's on the critical path" helps the agent make better trade-off decisions.
02

Use /goal for Complex, Long-Running Tasks

For big features or overnight tasks, use the /goal slash command. This puts Antigravity in a thorough, persistent mode where it won't stop until the goal is fully achieved — running tests, fixing errors, and iterating automatically.

💡 Tip: Use /goal for tasks like "Build a complete authentication system with registration, login, password reset, and email verification."
03

Leverage the Planning Mode

For complex changes, Antigravity enters Planning Mode before writing any code. It researches, analyses your codebase, and proposes a detailed implementation plan. Always review and approve the plan before proceeding — this is your chance to catch misunderstandings early.

💡 Tip: Ask clarifying questions in the plan review phase. "Can we also handle X edge case?" or "I prefer approach B" are all valid feedbacks.
04

Teach It Your Patterns with /learn

When Antigravity does something wrong or you correct it, use /learn to persist that correction as a rule. Over time, it adapts to your exact coding style, naming conventions, and architectural preferences.

💡 Tip: After you correct a repeated mistake (e.g., "always use our custom Button component, not plain <button>"), use /learn to lock that in permanently.
05

Use @mention to Reference Past Work

You can reference a previous conversation or context using @mention. This lets Antigravity pick up exactly where it left off, carry forward decisions from past sessions, and build on prior work without repeating yourself.

💡 Tip: After finishing a feature session, ask Antigravity to write a brief summary. Reference that summary in your next session for instant context loading.
06

Combine Skills for Maximum Power

Antigravity supports custom Skills — reusable instruction sets for specific tasks (e.g., "deploy to Vercel", "generate a data model"). Create skills for your most common workflows and activate them instantly with a single mention.

💡 Tip: Create a skill that encodes your entire component creation pattern — naming, file structure, test file, storybook story — and generate perfect components every time.
07

Schedule with /schedule

Use /schedule to set up recurring tasks or one-time timers. Antigravity can run a nightly code quality check, generate weekly progress reports, or remind you to run tests before a deployment window.

💡 Tip: Schedule a weekly /goal task to clean up tech debt — "Every Sunday, analyse the codebase for unused code, large functions, and missing tests."
08

Review All Proposed Commands

Antigravity will always ask your permission before running terminal commands, writing files, or making network requests. This is a safety feature — always read proposed commands before approving. You remain in control.

💡 Tip: Grant persistent permissions for commonly used safe commands (like git, npm, ls) to reduce friction without compromising safety.

Prompt Engineering for Antigravity

✅ DO
  • Provide specific file paths when referencing existing code
  • Describe the expected behaviour and edge cases
  • Specify which framework/library versions you're using
  • State constraints: "keep it under 100 lines", "no new dependencies"
  • Ask it to explain its decisions so you learn
  • Use /goal for complex multi-step tasks
  • Review plans carefully before approving
❌ DON'T
  • Give vague prompts: "make it better" or "fix the bug"
  • Approve commands without reading them
  • Ask for too many unrelated things in one message
  • Ignore the implementation plan phase

  • Use it as a replacement for understanding the code
  • Skip testing the generated code
Quick Reference

Developer
Cheat Sheet

The most important things to remember, distilled to the essentials.

⚡ Performance Checklist

  • Lighthouse score ≥ 90 on all metrics
  • LCP < 2.5s, FID < 100ms, CLS < 0.1
  • Images: WebP format, lazy loaded, proper srcset
  • Fonts: font-display: swap, preload key fonts
  • JS bundle < 150KB gzipped for initial load
  • Use service workers for offline support
  • Minimise render-blocking resources

🔒 Security Checklist

  • HTTPS everywhere, HSTS header set
  • Content-Security-Policy header configured
  • All user inputs sanitised server-side
  • No secrets in client-side code or git
  • Authentication tokens use httpOnly cookies
  • Rate limiting on all auth endpoints
  • Dependencies updated (npm audit passing)

♿ Accessibility Checklist

  • All images have descriptive alt text
  • Colour contrast ≥ 4.5:1 for normal text
  • Entire app keyboard-navigable
  • Focus indicators visible and clear
  • ARIA labels on all interactive elements
  • Screen reader tested (NVDA / VoiceOver)
  • No reliance on colour alone for meaning

🚀 Pre-Launch Checklist

  • Favicon and Open Graph meta tags set
  • robots.txt and sitemap.xml created
  • Error pages (404, 500) designed
  • Analytics and error monitoring active
  • All environment variables set in prod
  • Database backups configured
  • Rollback plan documented and tested

🤖 Antigravity Commands

  • /goal — Long-running autonomous task
  • /learn — Persist a correction or rule
  • /schedule — Set recurring or one-time tasks
  • /grill-me — Interactive planning interview
  • @mention — Reference past conversations
  • Planning Mode — Review before any code is written
  • Skills — Custom reusable instruction sets

📐 CSS Layout Quick Ref

  • display: flex → 1D layouts
  • display: grid → 2D layouts
  • box-sizing: border-box → always set globally
  • clamp(min, ideal, max) → fluid typography
  • gap → use instead of margins in flex/grid
  • min-content / max-content → intrinsic sizing
  • CSS Custom Properties → design tokens
